# ADR-0001: Планировщик расписаний профилей (модель Keenetic на Entware)

**Статус:** Accepted (реализовано и проверено на живом роутере 2026-06-10)
**Дата:** 2026-06-10
**Решает:** владелец проекта (Andrey)

## Контекст

Нужно управлять VPN+XOR-профилями по расписанию — как «Расписание работы» в родном
Keenetic (пользователь уже ротирует 4 родных OpenVPN-канала 3-часовыми окнами).
Изучено устройство оригинала через RCI (`show/sc/schedule`):

- расписание = список событий `{start|stop, час, минута, дни недели}`;
- события **edge-triggered**: NDM исполняет их в заданный момент, между событиями
  ручное управление интерфейсом работает;
- привязка — свойство интерфейса.

Ограничения нашей среды:
- наши `tun_invnet*` — чужие для NDM интерфейсы, его планировщик к ним не привязать;
- Entware: `crond` есть как бинарь, но не настроен (нет init-скрипта, нет crontab);
- источник истины о желаемом состоянии профиля — ключ `enabled` в
  `profile-meta/<name>.json`, исполнитель — `invnetctl` (enable/disable/reconcile);
- веб-панель обновляет список профилей каждые 5 секунд;
- jq на роутере без regex; busybox ash.

## Решение

1. **Хранение — per-profile**: ключ `schedule {enabled, actions[{do,hh,mm,dow[]}]}`
   внутри существующей меты профиля. Никаких отдельных именованных расписаний.
2. **Исполнитель — собственный демон** `init.d/S41invnet-sched`: на старте
   `invnetctl schedule-sync` (восстановление состояния), далее
   `schedule-tick` каждые 30 секунд.
3. **Семантика — как в Keenetic**: tick исполняет только события, чьё время попало
   в окно `(последний тик, сейчас]` (edge-triggered) → между событиями ручной тумблер
   работает. Полная синхронизация состояния — только при старте демона (перезагрузка)
   и после правки расписания.
4. **UI — недельная сетка** (часы 0–24 × Пн–Вс) под каждым профилем (аккордеон по ⏰);
   сетка — чистое представление: на клиенте конвертируется в события start/stop
   (границы закрашенных блоков) и обратно. Бэкенд хранит только события.

Вся арифметика времени — «минуты недели» по кругу (0..10079):
`ago = ((now − t) mod 10080)`, последнее событие = минимальный `ago`.

## Рассмотренные варианты

### Вариант A: собственный демон S41 + события в мете (принят)
| Измерение | Оценка |
|-----------|--------|
| Сложность | Низкая (~60 строк sh + 2 команды в invnetctl) |
| Зависимости | Нет (только jq, уже используется) |
| Точность | ±30 с — достаточно для VPN-окон |
| Знакомость паттерна | Высокая (init.d-демоны уже есть в проекте) |

**Плюсы:** самодостаточно, переживает перезагрузку (sync на старте), один механизм на все профили.
**Минусы:** ещё один фоновый процесс; точность ограничена интервалом тика.

### Вариант B: crond Entware + перегенерация crontab при правке
**Плюсы:** стандартный инструмент, минутная точность.
**Минусы:** crond на роутере не настроен (нет init/crontab — новая зависимость,
которую надо ставить и поддерживать в install.sh); crontab пришлось бы пересобирать
при каждой правке расписания; пропущенные события при перезагрузке всё равно требуют
отдельного sync-механизма — итого сложнее варианта A.

### Вариант C: регистрация tun_invnet в NDM, использование родного планировщика
**Минусы:** NDM не принимает чужие tun-интерфейсы в «Политику доступа»/расписания;
официального API регистрации из Entware нет. Нереализуемо (выяснено ранее при
работе с приоритетами ip rule).

### Вариант D: строгая синхронизация состояния каждым тиком (вместо edge-triggered)
**Плюсы:** проще рассуждать («состояние всегда = расписанию»).
**Минусы:** ручной тумблер «отскакивает» в течение 30 с — поведение хуже Keenetic;
отвергнут в пользу семантики оригинала.

## Анализ компромиссов

- A против B — главный выбор: **нулевые зависимости + единый sync-механизм** против
  «стандартности» cron. На одном устройстве с готовым паттерном init.d демон дешевле.
- Edge-triggered против строгого состояния: повторяем UX Keenetic (ручное управление
  между событиями), цена — состояние может расходиться с расписанием до следующего
  события. Это осознанное поведение, не баг.
- Сетка-в-события против хранения битовой маски: бэкенд не изменился вообще
  (события уже были реализованы и проверены в v1), минутная гранулярность БД
  сохранена (сетка просто пишет mm=0).

## Последствия

- Легче: автоматизация окон работы профилей без Keenetic; перенос на другой роутер
  (всё в архиве, без настройки cron); UI знакомый пользователю Keenetic.
- Сложнее: ещё один демон в составе (S41) — добавлен в install.sh/pack.sh и архивы.
- Пересмотреть, если: понадобится точность лучше ±30 с (уменьшить sleep или перейти
  на cron) или общие расписания на несколько профилей (вынести объекты расписаний
  из меты — миграция простая: ссылка вместо вложенного объекта).
